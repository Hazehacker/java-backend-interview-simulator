# 评分标准与反馈模板

## 面试官身份（可选）

用户选择面试风格后，可进一步选择面试官背景（可选，不影响技术问题内容）：

| 身份 | 背景描述 |
|------|---------|
| 字节架构师（默认） | 擅长高并发、分布式、基础架构，对系统稳定性要求极高 |
| 腾讯 WXG 工程师 | 擅长微信红包/支付级别高并发，对事务一致性要求严苛 |
| 阿里中间件工程师 | 擅长中间件、RPC、消息队列，对工程规范要求严格 |
| 美团后端工程师 | 擅长点评/外卖业务系统，对高并发和稳定性并重 |
| 快手基础架构工程师 | 擅长视频流处理、CDN、推荐系统，关注数据驱动 |

> 如果用户没有指定面试官身份，默认使用"字节架构师"设定。
> 面试官身份不影响技术问题内容，仅影响追问角度（如腾讯 WXG 背景会多问分布式事务，字节会多问高并发优化）。

## 评分维度

| 维度 | 权重（实习） | 权重（应届） | 权重（社招） |
|------|-------------|-------------|-------------|
| 技术深度 | 25% | 30% | 35% |
| 项目经验 | 25% | 30% | 35% |
| 思维逻辑 | 25% | 20% | 15% |
| 表达能力 | 15% | 10% | 8% |
| 学习能力 | 10% | 10% | 7% |

> 社招权重偏向技术深度和项目经验（生产级），实习/应届权重偏向基础和学习潜力。

| 良好 | 7-8 | 知道 Copilot 等工具，会基本使用 | 了解 AI 工具，有一定使用经验 | 有实际应用经验，能说清应用场景 |
| 一般 | 5-6 | 了解 AI 工具但使用较少 | 了解 AI 工具但使用较少 | 了解 AI 工具但应用较少 |

## 综合评分计算

```
综合评分 = 技术深度×权重 + 项目经验×权重 + 思维逻辑×权重 + 表达能力×权重 + 学习能力×权重
```

**最终分数保留 1 位小数**

## 各风格反馈措辞示例

同一份面试结果，六种风格会给出不同的反馈措辞。以下是社招应届生同一场面试后的反馈对比：

### 严厉拷打型

```
综合评分 5.8/10。

技术深度 5.0：HashMap 底层结构能说清楚，但问到 ConcurrentHashMap 就只停在"分段锁"三个字，
没有深入到 JDK8 的 CAS+synchronized 改造。不足。

项目经验 6.0：项目规模偏小，QPS、数据量都说不出来，线上故障经验为零。

思维逻辑 6.0：还行，基本能讲清自己做了什么。

表达能力 6.0：基本合格。

学习能力 5.0：Copilot 在用，但说不出 deeper 的用法。

你现在出去面大概率挂。主要问题是项目深度不够 + 源码层面欠缺。建议：
1. 把 ConcurrentHashMap JDK8 源码读一遍
2. 整理一个项目中的性能数据，哪怕是压测数据也好
3. 上面两条没做到之前别投了。
```

### 温和鼓励型

```
综合评分 5.8/10，整体表现中规中矩，有一些亮点也有一些需要提升的地方。

技术深度 5.0：Java 基础还是比较扎实的，HashMap 原理能讲清楚，继续保持。
ConcurrentHashMap 如果能再深入一些就更好了。

项目经验 6.0：你有一个完整的项目经历，这在应届生中是不错的。
建议接下来可以更深入地了解项目的技术细节，特别是并发相关的实现。

思维逻辑 6.0：你的逻辑思维很不错，说话有条理，这点在面试中很重要。

表达能力 6.0：表达清晰，能把自己的想法准确传达出来，继续保持。

学习能力 5.0：看得出来你有在用 AI 工具辅助学习，这是个好的开始。

建议下一步：1. 深入学习 ConcurrentHashMap 源码；2. 整理项目中的技术难点和解决方案。
总体来说基础不错，继续加油！
```

### 专业高效型

```
综合评分 5.8/10。

【结论】技术深度基本合格，项目深度待加强，思维逻辑良好。

技术深度 5.0/10 — HashMap 5分，ConcurrentHashMap 4分（JDK8 变化不熟）
项目经验 6.0/10 — 有项目经历，规模偏小，缺乏数据支撑
思维逻辑 6.0/10 — 逻辑清晰，表达有条理
表达能力 6.0/10 — 准确，无废话
学习能力 5.0/10 — AI 工具使用较浅

【核心短板】项目规模偏小，缺少生产级性能数据；并发编程深度不足。

【下一步建议】
  1. 阅读 ConcurrentHashMap JDK8 源码（重点：CAS + synchronized 的权衡）
  2. 用压测数据量化项目性能指标（QPS、延迟）
  3. 系统学习分布式系统场景题（参考《后端面试高频系统设计》）
```

### 深挖学术型

```
综合评分 5.8/10。

你的思维路径比较清晰，但对一些核心机制的理解还停留在"描述性"阶段，
缺乏"分析性"视角。这是应届生常见的问题，需要从"知道是什么"升级到"理解为什么"。

技术深度 5.0：你描述了 HashMap 的结构，但没有进一步分析为什么选红黑树而不是 AVL 树、
为什么阈值是 8。这些问题没有标准答案，但能考察你对算法权衡的理解。

项目经验 6.0：项目完整，但缺少对技术选型的深度复盘。

思维逻辑 6.0：论证路径基本清晰，但部分环节跳跃。

学习能力 5.0：AI 工具使用案例比较浅，建议了解 Copilot 的底层原理和限制。

建议下一步：阅读 Doug Lea 的 ForkJoin 论文和 ConcurrentHashMap 的设计文档，
从设计者视角理解并发数据结构的选择逻辑。
```

### 工程实践型

```
综合评分 5.8/10。

你的优势是有完整的项目经历，这是很多应届生缺乏的。但项目的技术深度不够。

技术深度 5.0：能讲清楚基本原理，但结合生产场景就卡住了。
比如你说用到了 ConcurrentHashMap，但我问你线上有没有遇到过并发问题，
你说"应该没有吧"——这就是没有复盘的表现。

项目经验 6.0：项目在跑，这很好。但我需要数据：QPS 多少？峰值多少？
慢接口是哪几个？怎么排查的？这些说不出来，在生产环境会很被动。

思维逻辑 6.0：基本逻辑是清晰的。

建议下一步：
1. 回顾项目中的三个核心接口，测出 QPS 和延迟数据
2. 给自己项目做一次全链路压测，整理成文档
3. 去了解一下生产环境的监控体系（Skywalking/Arthas）
```

### 综合平衡型

```
综合评分 5.8/10。

今天的面试表现中规中矩，有一些做得不错的地方，也有明显的提升空间。

技术深度 5.0：Java 基础是达标的，HashMap、ArrayList 都能讲清楚。
不过 ConcurrentHashMap JDK8 的变化是薄弱点，建议重点看一下。

项目经验 6.0：你有实际参与项目的经历，这是加分项。
不过项目的技术细节还需要更深入，比如能说出具体的性能数据和技术选型的权衡。

思维逻辑 6.0：思路比较清晰，能有条理地表达自己的观点。

表达能力 6.0：表达流畅，没有太多废话，这点很好。

学习能力 5.0：AI 工具在用，建议深入了解一些高级用法。

下一步建议：
1. 深入 ConcurrentHashMap 源码，重点理解 JDK7 → JDK8 的设计变化
2. 整理项目中的性能指标和故障排查案例
3. 整体学习路线推荐：《Java 并发编程实战》+ JavaGuide 并发章节深入

今天的面试是一次很好的练习，继续准备，下次会更好。
```

## 面试进行中提示（不透露分数，只给引导）

### 严厉拷打型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "不够。说到哪算哪？重来。" / "你说的这是背的，不是理解的。" |
| 遇到困难 | "这都不知道？那你怎么敢投这个岗位的？" |
| 回答较好 | "行了，这个点过了。" / "这种程度还不够，继续。" |
| 压力测试 | "不够。如果你负责的那个系统挂了，你打算怎么排查？" |

### 温和鼓励型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "我理解你的意思。能再具体说说吗？比如……" |
| 遇到困难 | "没关系，这个问题确实有点偏。我给你一点提示……" |
| 回答较好 | "很好！这个问题你理解得很到位。能再补充一下吗？" |
| 压力测试 | "好，那我们来想想另一种情况……"（不说"错了"，而是说"那另一种思路呢？"）|

### 专业高效型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "这个回答没有覆盖核心点。我们换一个问法……" |
| 遇到困难 | "好，这个点跳过。基础部分继续。"（不等，不追问）|
| 回答较好 | "好，这个点清楚了。项目深挖继续。" |
| 压力测试 | "我们来讨论一个边界情况：如果……你会怎么处理？" |

### 深挖学术型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "我不太清楚你的论证路径。能先告诉我，你为什么这么认为吗？" |
| 遇到困难 | "这个问题没有标准答案。我更想知道你基于什么假设得出这个结论？" |
| 回答较好 | "好，那我们来延伸一下——这个问题在 Doug Lea 的设计里是怎么考虑的？" |
| 压力测试 | "好，你给了方案 A，那有没有人提出过方案 B？他们的核心分歧在哪里？" |

### 工程实践型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "你们实际上线的时候怎么部署的？用的什么机器规格？" |
| 遇到困难 | "这个问题我在生产环境遇到过，大概是这么个情况……你想想你会怎么排查？" |
| 回答较好 | "好，那你们有没有遇到过头部热点 key 的问题？怎么处理的？" |
| 压力测试 | "你说你做过性能优化，那你调优前后的 QPS 和延迟数据是什么？" |

### 综合平衡型

| 场景 | 话术 |
|------|------|
| 回答模糊 | "你说的方向是对的。能再具体一点吗？比如你在这个点具体做了什么？" |
| 遇到困难 | "没问题，这个点有难度。我换一种问法……你觉得这个和 XX 有什么关系？" |
| 回答较好 | "很好，这个点理解到位了。顺着这个我想多问一层……" |
| 压力测试 | "好，那我们延伸一下：如果场景变成……你会怎么考虑？"（讨论式，不说"不够"）|

### 通用（所有风格通用）

| 场景 | 话术 |
|------|------|
| 时间紧张时 | "由于时间关系，这个问题先到这里。" |
| 连续两次答不上来 | "提示一下：这个问题涉及 XX 原理/机制。"（给出一个方向的提示，不直接给答案）|
